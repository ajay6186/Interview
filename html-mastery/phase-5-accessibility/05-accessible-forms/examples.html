Accessible Forms — 50 Examples

1. Label with for/id association

<label for="username">Username</label>
<input id="username" type="text" name="username">

2. Label wrapping the input (implicit association)

<label>
  Username
  <input type="text" name="username">
</label>

3. aria-label — when no visible label exists

<input type="search" aria-label="Search the site" name="q">

4. aria-labelledby — label from existing element

<h2 id="billing-heading">Billing Address</h2>
<input aria-labelledby="billing-heading" type="text" name="street">

5. Placeholder is NOT a label

<!-- BAD: placeholder disappears on input -->
<input type="email" placeholder="email@example.com">

<!-- GOOD: visible label + placeholder as hint -->
<label for="email">Email</label>
<input id="email" type="email" placeholder="email@example.com">

6. Marking fields as required

<label for="name">Full name <span aria-hidden="true">*</span></label>
<input id="name" type="text" required aria-required="true">
<p><span aria-hidden="true">*</span> Required fields</p>

7. aria-required vs required attribute

<!-- HTML5 required: browser validates and blocks submit -->
<input required>

<!-- aria-required: informational only for screen readers -->
<input aria-required="true">

<!-- Use both for maximum compatibility -->
<input required aria-required="true">

8. Hint text with aria-describedby

<label for="password">Password</label>
<input id="password" type="password" aria-describedby="pw-hint">
<p id="pw-hint">At least 8 characters, including a number.</p>

9. Multiple descriptors — aria-describedby with list

<input id="email"
       aria-describedby="email-hint email-error"
       type="email">
<p id="email-hint">We'll send a confirmation link.</p>
<p id="email-error" role="alert" hidden>Please enter a valid email.</p>

10. Inline error message with role="alert"

<span id="name-error" role="alert" aria-live="polite"></span>

<script>
  function showError(msg) {
    document.getElementById('name-error').textContent = msg;
  }
</script>

11. aria-invalid to mark invalid fields

<input id="email" type="email"
       aria-invalid="true"
       aria-describedby="email-error">
<span id="email-error" role="alert">Invalid email address.</span>

12. aria-invalid states

aria-invalid="false"    — field is valid (default)
aria-invalid="true"     — field is invalid
aria-invalid="grammar"  — grammar error
aria-invalid="spelling" — spelling error

13. Reset aria-invalid after correction

input.addEventListener('input', () => {
  if (input.validity.valid) {
    input.setAttribute('aria-invalid', 'false');
    errorEl.textContent = '';
  }
});

19. Form validation — show errors on submit

form.addEventListener('submit', e => {
  const errors = validate(form);
  if (errors.length) {
    e.preventDefault();
    displayErrors(errors);
    errors[0].field.focus(); // move focus to first invalid field
  }
});

20. Error summary at top of form

<div id="error-summary" role="alert" aria-live="assertive" hidden>
  <h2>Please fix the following errors:</h2>
  <ul id="error-list"></ul>
</div>
Link each error in the summary to the relevant field with <a href="#field-id">.

21. Error summary with links

function showSummary(errors) {
  const list = document.getElementById('error-list');
  list.innerHTML = errors.map(e =>
    `<li><a href="#${e.id}">${e.message}</a></li>`
  ).join('');
  const summary = document.getElementById('error-summary');
  summary.hidden = false;
  summary.focus();
}

26. Button type — always specify type

<!-- Submit (default when no type specified inside a form) -->
<button type="submit">Submit</button>

<!-- Reset form -->
<button type="reset">Clear</button>

<!-- No form action -->
<button type="button" onclick="doSomething()">Action</button>

27. Accessible loading state on submit

form.addEventListener('submit', async e => {
  e.preventDefault();
  submitBtn.setAttribute('aria-busy', 'true');
  submitBtn.textContent = 'Saving...';
  await saveData();
  submitBtn.removeAttribute('aria-busy');
  submitBtn.textContent = 'Save';
});

46. Form accessibility checklist

// ✓ Every input has a visible label
// ✓ Required fields are indicated textually
// ✓ Error messages are linked via aria-describedby
// ✓ Errors appear as role="alert" or aria-live
// ✓ Focus moves to first error on submit
// ✓ Success confirmation is announced
// ✓ Fieldsets group related controls

47. Testing with screen reader

// Manual test steps:
// 1. Tab to each field — is the label announced?
// 2. Are required fields identified?
// 3. Submit empty form — are errors announced?
// 4. Are error messages linked to the fields?
// 5. Is success confirmed after valid submission?
