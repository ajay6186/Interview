Examples 5.1 — ARIA Basics

BASIC (1–13)

1. aria-label on a button with no visible text

<button aria-label="Close dialog">
  <svg aria-hidden="true" focusable="false">
    <use href="#icon-x"></use>
  </svg>
</button>

When a button contains only an icon, aria-label provides the accessible name that screen readers announce. The SVG is hidden from the accessibility tree with aria-hidden.

2. Icon button — search

<button aria-label="Search">
  <svg aria-hidden="true" focusable="false" width="16" height="16">
    <path d="M11 11l4 4M7 13A6 6 0 107 1a6 6 0 000 12z"/>
  </svg>
</button>

<!-- Alternative: visually hidden span -->
<button>
  <svg aria-hidden="true" focusable="false">...</svg>
  <span class="sr-only">Search</span>
</button>

Two equivalent patterns for naming icon-only buttons. The sr-only (visually hidden) class approach keeps text in the DOM for translation tools.

3. aria-labelledby referencing a visible heading

<section aria-labelledby="section-heading">
  <h2 id="section-heading">Recent Orders</h2>
  <ul>
    <li>Order #1042 — Shipped</li>
    <li>Order #1041 — Delivered</li>
  </ul>
</section>

aria-labelledby points to the ID of an existing visible element. The section's accessible name becomes "Recent Orders". Prefer this over aria-label when the label text is already visible.

4. aria-describedby for supplementary hint text

<label for="password">Password</label>
<input
  type="password"
  id="password"
  aria-describedby="password-hint"
/>
<p id="password-hint">
  Must be at least 8 characters with one uppercase letter.
</p>

aria-describedby links the input to a description element. Screen readers announce the label first, then (after a pause) the description. Unlike aria-labelledby, it adds supplementary—not primary—information.

5. aria-hidden="true" on purely decorative elements

<!-- Decorative divider -->
<hr aria-hidden="true">

<!-- Decorative icon beside labeled text -->
<span aria-hidden="true">★</span>
<span>Featured</span>

<!-- Decorative image -->
<img src="wave.png" alt="" aria-hidden="true">

aria-hidden="true" removes an element and its subtree from the accessibility tree entirely. Use it for decorative icons, repeated separators, and visual flourishes that add no information.

6. role="presentation" to remove implicit semantics

<!-- Layout table — not data -->
<table role="presentation">
  <tr>
    <td><img src="logo.png" alt="Acme Inc"></td>
    <td><nav>...</nav></td>
  </tr>
</table>

<!-- List used only for visual spacing -->
<ul role="presentation">
  <li>Item A</li>
  <li>Item B</li>
</ul>

role="presentation" strips an element's native role from the accessibility tree while keeping it visible. Child elements retain their own roles. Equivalent to role="none".

7. role="none" — modern alias for presentation

<!-- Preferred modern form -->
<div role="none">
  <img src="divider.svg" alt="">
</div>

<!-- Both are identical; "none" is listed first in ARIA 1.2 -->
<li role="none"><a href="/home">Home</a></li>

role="none" is the preferred modern alias for role="presentation". The li element's list-item role is removed so screen readers only encounter the link, which is a common pattern for navigation lists.

8. aria-hidden on icon font spans

<button>
  <i class="fa fa-trash" aria-hidden="true"></i>
  Delete item
</button>

<!-- Without aria-hidden, some screen readers announce
     the Unicode character used by the icon font -->
<button>
  <i class="fa fa-envelope" aria-hidden="true"></i>
  Send email
</button>

Icon fonts insert characters via CSS :before pseudo-elements. Without aria-hidden="true", some AT may announce the underlying Unicode private-use character. Always hide icon-font elements.

9. When NOT to use ARIA — first rule of ARIA

<!-- BAD: reinventing a native button -->
<div role="button" tabindex="0" onclick="submit()">Submit</div>

<!-- GOOD: use the native element -->
<button type="submit">Submit</button>

<!-- BAD: adding redundant role -->
<button role="button">Click me</button>

<!-- GOOD: native semantics are already correct -->
<button>Click me</button>

The first rule of ARIA: don't use ARIA if a native HTML element already provides the semantics. Native elements have built-in keyboard behavior, role, and state that ARIA-on-div lacks.

10. aria-label on form inputs

<!-- When a visible label would be redundant (e.g. table cell inputs) -->
<table>
  <thead>
    <tr><th>Item</th><th>Quantity</th></tr>
  </thead>
  <tbody>
    <tr>
      <td>Widget A</td>
      <td>
        <input type="number" aria-label="Quantity for Widget A" value="1">
      </td>
    </tr>
  </tbody>
</table>

aria-label is appropriate when a visible label would be redundant or impractical, such as quantity fields in a table where the column header and row header together provide context visually.

11. aria-expanded for accordion disclosure

<button
  aria-expanded="false"
  aria-controls="faq-1-body"
  id="faq-1-btn"
>
  What is your return policy?
</button>
<div id="faq-1-body" hidden>
  <p>You may return items within 30 days...</p>
</div>

<!-- When opened, JS sets: -->
<!-- button.setAttribute('aria-expanded', 'true') -->
<!-- div.removeAttribute('hidden') -->

aria-expanded conveys the open/closed state of a controlled region. Pair it with aria-controls pointing to the controlled element's ID. Toggle both the attribute and the hidden state in JS.

12. aria-controls for widget relationships

<!-- Dropdown that controls a listbox -->
<button
  id="sort-btn"
  aria-controls="sort-list"
  aria-expanded="false"
  aria-haspopup="listbox"
>
  Sort by: Relevance
</button>
<ul id="sort-list" role="listbox" aria-labelledby="sort-btn" hidden>
  <li role="option" aria-selected="true">Relevance</li>
  <li role="option" aria-selected="false">Price: Low to High</li>
  <li role="option" aria-selected="false">Newest</li>
</ul>

aria-controls identifies the element(s) a widget controls. It is an IDREF relationship — both elements must be in the same document. Support varies; always pair with visible/hidden state changes.

13. aria-required on form fields

<form>
  <label for="email">
    Email address
    <span aria-hidden="true"> *</span>
  </label>
  <input
    type="email"
    id="email"
    name="email"
    aria-required="true"
    autocomplete="email"
  >
  <p><span aria-hidden="true">*</span> Required field</p>
</form>

aria-required="true" informs AT that the field must be filled before submission. The native required attribute also adds browser validation; use both. The asterisk is decorative, so it is hidden from AT with aria-hidden.

INTERMEDIATE (14–26)

14. aria-checked for a custom checkbox

<div
  role="checkbox"
  aria-checked="false"
  tabindex="0"
  id="custom-cb"
  aria-labelledby="cb-label"
>
  <svg aria-hidden="true">...</svg>
</div>
<span id="cb-label">Subscribe to newsletter</span>

<script>
  const cb = document.getElementById('custom-cb');
  cb.addEventListener('keydown', e => {
    if (e.key === ' ') {
      e.preventDefault();
      const checked = cb.getAttribute('aria-checked') === 'true';
      cb.setAttribute('aria-checked', String(!checked));
    }
  });
  cb.addEventListener('click', () => {
    const checked = cb.getAttribute('aria-checked') === 'true';
    cb.setAttribute('aria-checked', String(!checked));
  });
</script>

When building a custom checkbox, you must implement role="checkbox", aria-checked state management, tabindex="0", and keyboard handling (Space to toggle). Prefer native <input type="checkbox"> whenever possible.

15. aria-checked for tri-state (mixed)

<!-- Parent checkbox with indeterminate state -->
<div
  role="checkbox"
  aria-checked="mixed"
  tabindex="0"
  id="select-all"
>
  Select all
</div>

<!-- In JS, set aria-checked to:
     "true"  — all children checked
     "false" — no children checked
     "mixed" — some children checked -->

aria-checked="mixed" represents an indeterminate state. This is used for "select all" controls when only some subordinate items are selected. Native <input> achieves this via the indeterminate JS property.

16. aria-selected for a tab panel widget

<div role="tablist" aria-label="Product details">
  <button role="tab" aria-selected="true"  aria-controls="panel-desc" id="tab-desc">Description</button>
  <button role="tab" aria-selected="false" aria-controls="panel-spec" id="tab-spec" tabindex="-1">Specs</button>
  <button role="tab" aria-selected="false" aria-controls="panel-rev"  id="tab-rev"  tabindex="-1">Reviews</button>
</div>
<div role="tabpanel" id="panel-desc" aria-labelledby="tab-desc">...</div>
<div role="tabpanel" id="panel-spec" aria-labelledby="tab-spec" hidden>...</div>
<div role="tabpanel" id="panel-rev"  aria-labelledby="tab-rev"  hidden>...</div>

aria-selected="true" marks the active tab. Non-selected tabs use tabindex="-1" so only one tab is in the natural tab order (roving tabindex pattern). Arrow keys navigate between tabs.

17. aria-selected for a listbox option

<ul role="listbox" aria-label="Choose a color" aria-multiselectable="false">
  <li role="option" aria-selected="false">Red</li>
  <li role="option" aria-selected="true">Green</li>
  <li role="option" aria-selected="false">Blue</li>
</ul>

Within a listbox, aria-selected conveys which option is chosen. For multi-select listboxes, add aria-multiselectable="true" and allow multiple options to have aria-selected="true".

18. aria-disabled vs the disabled attribute

<!-- Native disabled: removed from tab order, not focusable -->
<button disabled>Submit</button>

<!-- aria-disabled: focusable but announced as disabled -->
<button aria-disabled="true" onclick="return false;">
  Submit
</button>

<!-- Use aria-disabled when you want the element to remain
     focusable — e.g., so users can hear why it is disabled -->
<button
  aria-disabled="true"
  aria-describedby="submit-why-disabled"
  onclick="return false;"
>Submit</button>
<p id="submit-why-disabled">Complete all required fields to enable.</p>

aria-disabled="true" marks a widget as disabled without removing it from the tab order. This is useful when you want keyboard users to reach the element and hear the associated description explaining why it is disabled.

19. aria-readonly on a read-only field

<label for="username">Username</label>
<input
  type="text"
  id="username"
  value="jsmith"
  readonly
  aria-readonly="true"
  aria-describedby="username-note"
>
<p id="username-note">
  Username cannot be changed after registration.
</p>

aria-readonly="true" signals that the field value cannot be modified. The native readonly attribute already conveys this for inputs, but aria-readonly is needed for custom editable widgets (e.g., contenteditable divs styled as inputs).

20. aria-invalid with aria-errormessage

<label for="phone">Phone number</label>
<input
  type="tel"
  id="phone"
  aria-invalid="true"
  aria-errormessage="phone-error"
  aria-describedby="phone-error"
  value="abc"
>
<p id="phone-error" role="alert">
  Please enter a valid phone number (digits only).
</p>

<!-- aria-invalid values: "true" | "false" | "grammar" | "spelling" -->

aria-invalid="true" flags the field as having an invalid value. aria-errormessage points to the ID of the error description. Using role="alert" on the error paragraph ensures immediate announcement when it appears.

21. aria-pressed for toggle buttons

<button aria-pressed="false" id="bold-btn">
  <svg aria-hidden="true">...</svg>
  Bold
</button>

<script>
  const btn = document.getElementById('bold-btn');
  btn.addEventListener('click', () => {
    const pressed = btn.getAttribute('aria-pressed') === 'true';
    btn.setAttribute('aria-pressed', String(!pressed));
    // Apply/remove bold formatting
  });
</script>

<!-- Screen reader announces: "Bold, toggle button, not pressed" -->
<!-- After click:             "Bold, toggle button, pressed"     -->

aria-pressed turns a button into a toggle button with three possible states: "true" (pressed/active), "false" (not pressed), and "mixed" (partially applied, e.g., mixed bold in a text selection).

22. aria-current="page" for active nav link

<nav aria-label="Main">
  <ul>
    <li><a href="/"      aria-current="page">Home</a></li>
    <li><a href="/about">About</a></li>
    <li><a href="/blog"> Blog</a></li>
  </ul>
</nav>

<!-- Other aria-current values:
     "step"     — current step in a wizard
     "date"     — current date in a calendar
     "location" — current location in a map/flowchart
     "true"     — generic current item -->

aria-current="page" marks the link corresponding to the currently loaded page. Screen readers announce "current" alongside the link name. Set it dynamically in SPAs when the route changes.

23. aria-haspopup for menus and dialogs

<!-- Menu button -->
<button aria-haspopup="menu" aria-expanded="false" aria-controls="user-menu">
  My Account
</button>
<ul role="menu" id="user-menu" hidden>
  <li role="menuitem"><a href="/profile">Profile</a></li>
  <li role="menuitem"><a href="/settings">Settings</a></li>
  <li role="menuitem"><button>Sign out</button></li>
</ul>

<!-- aria-haspopup values:
     "menu"    | "listbox" | "tree"
     "grid"    | "dialog"  | "true" (= menu) -->

aria-haspopup tells AT that activating the element will open a popup of the specified type. This primes the user to expect a secondary layer. The value should match the role of the popup container.

24. aria-owns for non-hierarchical DOM relationships

<!-- A tree node owns its sub-tree, even if the sub-tree
     is elsewhere in the DOM (e.g., rendered in a portal) -->
<li
  role="treeitem"
  aria-owns="subtree-42"
  aria-expanded="true"
>
  Documents
</li>

<!-- Elsewhere in the DOM (e.g., portal container): -->
<ul role="group" id="subtree-42">
  <li role="treeitem">Resume.pdf</li>
  <li role="treeitem">Cover Letter.docx</li>
</ul>

aria-owns creates a parent-child relationship in the accessibility tree that does not exist in the DOM. Use it sparingly — only when DOM reordering is impossible. It can cause confusion if misused.

25. aria-activedescendant for composite widgets

<!-- The container holds focus; aria-activedescendant tracks
     the currently "active" child visually -->
<ul
  role="listbox"
  tabindex="0"
  id="fruit-list"
  aria-activedescendant="opt-banana"
  aria-label="Favourite fruit"
>
  <li role="option" id="opt-apple"  aria-selected="false">Apple</li>
  <li role="option" id="opt-banana" aria-selected="true">Banana</li>
  <li role="option" id="opt-cherry" aria-selected="false">Cherry</li>
</ul>

<!-- On arrow key, JS updates aria-activedescendant to the new option's ID -->

aria-activedescendant is used in "managed focus" composites. The container element holds real DOM focus; AT follows aria-activedescendant to know which child is currently active, avoiding focus-thrashing on every keystroke.

26. aria-valuenow / aria-valuemin / aria-valuemax / aria-valuetext for sliders

<div
  role="slider"
  tabindex="0"
  aria-label="Volume"
  aria-valuenow="60"
  aria-valuemin="0"
  aria-valuemax="100"
  aria-valuetext="60 percent"
  id="volume-slider"
>
  <div class="thumb" style="left: 60%"></div>
</div>

<script>
  // On arrow key, update aria-valuenow and aria-valuetext
  slider.setAttribute('aria-valuenow', newValue);
  slider.setAttribute('aria-valuetext', newValue + ' percent');
</script>

Custom sliders need all four value attributes. aria-valuetext overrides the numeric announcement with a human-friendly string — essential when the raw number is meaningless (e.g., "Low / Medium / High" positions).

NESTED (27–38)

27. Complete tab widget ARIA pattern

<!-- Tab list -->
<div role="tablist" aria-label="Settings sections">
  <button
    role="tab"
    id="tab-general"
    aria-selected="true"
    aria-controls="panel-general"
  >General</button>
  <button
    role="tab"
    id="tab-security"
    aria-selected="false"
    aria-controls="panel-security"
    tabindex="-1"
  >Security</button>
  <button
    role="tab"
    id="tab-billing"
    aria-selected="false"
    aria-controls="panel-billing"
    tabindex="-1"
  >Billing</button>
</div>

<!-- Tab panels -->
<div
  role="tabpanel"
  id="panel-general"
  aria-labelledby="tab-general"
  tabindex="0"
>
  <h2>General Settings</h2>
  <!-- content -->
</div>
<div
  role="tabpanel"
  id="panel-security"
  aria-labelledby="tab-security"
  tabindex="0"
  hidden
>
  <!-- content -->
</div>
<div
  role="tabpanel"
  id="panel-billing"
  aria-labelledby="tab-billing"
  tabindex="0"
  hidden
>
  <!-- content -->
</div>

Full tab widget: tablist container, tab buttons with aria-selected and aria-controls, tabpanel with aria-labelledby and tabindex="0" so panel content is reachable. Non-active tabs use tabindex="-1"; arrow keys cycle through them.

28. Accordion pattern with ARIA

<div class="accordion">
  <!-- Item 1 -->
  <h3>
    <button
      type="button"
      aria-expanded="true"
      aria-controls="acc-body-1"
      id="acc-hd-1"
    >
      Shipping & Delivery
    </button>
  </h3>
  <div
    id="acc-body-1"
    role="region"
    aria-labelledby="acc-hd-1"
  >
    <p>Standard shipping takes 5-7 business days...</p>
  </div>

  <!-- Item 2 -->
  <h3>
    <button
      type="button"
      aria-expanded="false"
      aria-controls="acc-body-2"
      id="acc-hd-2"
    >
      Returns & Refunds
    </button>
  </h3>
  <div
    id="acc-body-2"
    role="region"
    aria-labelledby="acc-hd-2"
    hidden
  >
    <p>Return within 30 days for a full refund...</p>
  </div>
</div>

The WAI-ARIA accordion pattern: heading > button with aria-expanded, paired panel div with role="region" and aria-labelledby. The role="region" creates a landmark only when the section has a label, enabling screen reader landmark navigation.

29. Dropdown menu ARIA pattern

<div class="menu-wrapper">
  <button
    type="button"
    id="nav-menu-btn"
    aria-haspopup="menu"
    aria-expanded="false"
    aria-controls="nav-menu"
  >
    Products
    <svg aria-hidden="true"><!-- chevron --></svg>
  </button>

  <ul
    role="menu"
    id="nav-menu"
    aria-labelledby="nav-menu-btn"
    hidden
  >
    <li role="none">
      <a role="menuitem" href="/widgets">Widgets</a>
    </li>
    <li role="none">
      <a role="menuitem" href="/gadgets">Gadgets</a>
    </li>
    <li role="none">
      <button role="menuitem">Compare all</button>
    </li>
  </ul>
</div>

Menu pattern: button with aria-haspopup="menu" and aria-expanded, ul with role="menu", li with role="none", and links/buttons with role="menuitem". Arrow keys navigate items; Escape closes and returns focus to the button.

30. Dialog with ARIA

<div
  role="dialog"
  id="confirm-dialog"
  aria-modal="true"
  aria-labelledby="dialog-title"
  aria-describedby="dialog-desc"
>
  <h2 id="dialog-title">Confirm deletion</h2>
  <p id="dialog-desc">
    This will permanently delete your account and all data.
    This action cannot be undone.
  </p>
  <button type="button" id="dialog-cancel">Cancel</button>
  <button type="button" id="dialog-confirm">Delete account</button>
</div>

<!-- On open:
     - Move focus to first focusable element (Cancel button)
     - Trap Tab/Shift+Tab within dialog
     - Escape key closes dialog and restores focus -->

Dialogs need role="dialog", aria-modal="true" (tells AT to ignore content outside), aria-labelledby (title), and aria-describedby (description). On open, focus must move into the dialog; on close, it must return to the trigger.

31. Listbox pattern

<label id="lang-label">Preferred language</label>
<ul
  role="listbox"
  aria-labelledby="lang-label"
  tabindex="0"
  aria-activedescendant="opt-en"
>
  <li role="option" id="opt-en" aria-selected="true">English</li>
  <li role="option" id="opt-fr" aria-selected="false">French</li>
  <li role="option" id="opt-de" aria-selected="false">German</li>
  <li role="option" id="opt-ja" aria-selected="false">Japanese</li>
</ul>

<!-- Keyboard: Arrow keys move aria-activedescendant,
               Enter/Space selects (sets aria-selected="true") -->

A listbox is a scrollable selection widget. The container holds focus via tabindex="0" and aria-activedescendant. Each option has role="option" and aria-selected. This pattern is used when a native <select> cannot be styled sufficiently.

32. Combobox basics (ARIA 1.2 pattern)

<label for="city-input">City</label>
<div class="combobox-wrapper">
  <input
    type="text"
    id="city-input"
    role="combobox"
    aria-autocomplete="list"
    aria-expanded="false"
    aria-controls="city-listbox"
    aria-activedescendant=""
    autocomplete="off"
  >
  <ul
    role="listbox"
    id="city-listbox"
    aria-label="Cities"
    hidden
  >
    <li role="option" id="opt-nyc">New York</li>
    <li role="option" id="opt-la">Los Angeles</li>
    <li role="option" id="opt-chi">Chicago</li>
  </ul>
</div>

The ARIA 1.2 combobox pattern places role="combobox" on the input itself (not a wrapper div). aria-autocomplete="list" means suggestions are shown in a popup list. aria-expanded and aria-activedescendant update as the user types.

33. Tooltip pattern

<button
  type="button"
  aria-describedby="tt-copy"
  id="copy-btn"
>
  <svg aria-hidden="true">...</svg>
  <span class="sr-only">Copy</span>
</button>

<div
  role="tooltip"
  id="tt-copy"
  hidden
>
  Copy to clipboard
</div>

<!-- Show tooltip on focus/hover; hide on blur/mouseleave -->
<!-- Tooltip is supplementary; the button already has "Copy" -->

role="tooltip" marks descriptive popups that appear on hover/focus. Link the trigger with aria-describedby. The tooltip should not contain interactive elements and should be dismissible with Escape (WCAG 1.4.13). It is supplementary — not the primary label.

34. Tree widget

<ul role="tree" aria-label="File browser">
  <li
    role="treeitem"
    aria-expanded="true"
    aria-selected="false"
  >
    <span>src</span>
    <ul role="group">
      <li role="treeitem" aria-selected="true">
        <span>index.ts</span>
      </li>
      <li
        role="treeitem"
        aria-expanded="false"
        aria-selected="false"
      >
        <span>components</span>
        <ul role="group">
          <li role="treeitem" aria-selected="false">
            <span>Button.tsx</span>
          </li>
        </ul>
      </li>
    </ul>
  </li>
</ul>

Tree widgets use role="tree" on the root, role="treeitem" on each node, and role="group" for child subtrees. Expandable nodes need aria-expanded. Keyboard: arrows navigate, Enter/Space select or expand/collapse.

35. Grid widget

<table
  role="grid"
  aria-label="Task list"
  aria-rowcount="100"
  aria-colcount="4"
>
  <thead>
    <tr role="row">
      <th role="columnheader" aria-sort="ascending">Title</th>
      <th role="columnheader" aria-sort="none">Status</th>
      <th role="columnheader" aria-sort="none">Priority</th>
      <th role="columnheader">Actions</th>
    </tr>
  </thead>
  <tbody>
    <tr role="row" aria-rowindex="1">
      <td role="gridcell">Design review</td>
      <td role="gridcell">In progress</td>
      <td role="gridcell">High</td>
      <td role="gridcell"><button>Edit</button></td>
    </tr>
  </tbody>
</table>

role="grid" turns a table into an interactive widget where cells can receive focus and contain interactive elements. aria-sort indicates the sort direction on column headers. Arrow keys navigate cells; Tab moves to next interactive element.

36. Custom star rating widget

<div
  role="radiogroup"
  aria-label="Product rating"
  aria-required="true"
>
  <span>Rate this product:</span>
  <button role="radio" aria-checked="false" aria-label="1 star">★</button>
  <button role="radio" aria-checked="false" aria-label="2 stars">★</button>
  <button role="radio" aria-checked="true"  aria-label="3 stars">★</button>
  <button role="radio" aria-checked="false" aria-label="4 stars">★</button>
  <button role="radio" aria-checked="false" aria-label="5 stars">★</button>
</div>

Star rating widgets group radio-like controls with role="radiogroup". Each star uses role="radio" with aria-checked. aria-label on each star gives a meaningful name ("3 stars") rather than the visual character "★".

37. Spinbutton (numeric stepper)

<label id="qty-label">Quantity</label>
<div
  role="spinbutton"
  aria-labelledby="qty-label"
  aria-valuenow="1"
  aria-valuemin="1"
  aria-valuemax="99"
  tabindex="0"
  id="qty-spin"
>
  <button
    aria-label="Decrease quantity"
    tabindex="-1"
    id="qty-dec"
  >−</button>
  <span aria-hidden="true">1</span>
  <button
    aria-label="Increase quantity"
    tabindex="-1"
    id="qty-inc"
  >+</button>
</div>

role="spinbutton" is for widgets that let users select a value from a numeric range. The container holds focus; Up/Down arrows increment/decrement. The inner buttons are tabindex="-1" so only one tab stop exists.

38. role="button" on a div — full keyboard implementation

<div
  role="button"
  tabindex="0"
  id="custom-btn"
  aria-pressed="false"
  class="custom-toggle"
>
  Enable notifications
</div>

<script>
  const btn = document.getElementById('custom-btn');

  btn.addEventListener('click', toggle);
  btn.addEventListener('keydown', e => {
    // buttons activate on both Enter and Space
    if (e.key === 'Enter' || e.key === ' ') {
      e.preventDefault(); // prevent page scroll on Space
      toggle();
    }
  });

  function toggle() {
    const pressed = btn.getAttribute('aria-pressed') === 'true';
    btn.setAttribute('aria-pressed', String(!pressed));
  }
</script>

When role="button" is used on a non-button element, you must manually add tabindex="0", click handler, AND keydown handler for both Enter and Space. Native <button> handles all this automatically — always prefer it.

45. ARIA in shadow DOM

<!-- Custom element with shadow DOM -->
<my-toggle id="notif-toggle"></my-toggle>

<!-- Inside the shadow root: -->
<template id="toggle-template">
  <button
    part="button"
    aria-pressed="false"
  >
    <slot>Toggle</slot>
  </button>
</template>

<!-- Cross-root ARIA (AOM / ARIA reflection) -->
<script>
  class MyToggle extends HTMLElement {
    connectedCallback() {
      const shadow = this.attachShadow({ mode: 'open' });
      // ARIA attributes on host element may not reach shadow DOM
      // Use element internals for proper cross-root ARIA
      this._internals = this.attachInternals();
      this._internals.ariaPressed = 'false';
    }
  }
  customElements.define('my-toggle', MyToggle);
</script>

Shadow DOM creates ARIA isolation problems. ARIA IDREFs cannot cross shadow boundaries. Use ElementInternals (attachInternals()) and ARIA reflection (element.ariaLabel, etc.) to expose accessibility state from within shadow roots.

47. aria-busy for loading states

<div
  id="search-results"
  aria-live="polite"
  aria-busy="false"
  aria-label="Search results"
>
  <!-- Results rendered here -->
</div>

<script>
  async function search(query) {
    const region = document.getElementById('search-results');

    // Signal loading — AT may wait before reading the region
    region.setAttribute('aria-busy', 'true');
    region.innerHTML = '<p>Loading...</p>';

    const results = await fetchResults(query);
    region.innerHTML = renderResults(results);

    // Signal complete — AT will now announce the updated content
    region.setAttribute('aria-busy', 'false');
  }
</script>

aria-busy="true" signals to AT that the region is being updated and they should wait before reading it. Set it before starting the update and clear it to "false" when complete. Useful for live regions that make multiple rapid changes.

50. ARIA label computation priority order

<!-- ARIA accessible name computation (highest → lowest priority): -->

<!-- 1. aria-labelledby (highest) -->
<input aria-labelledby="lbl" /><span id="lbl">Name</span>

<!-- 2. aria-label -->
<input aria-label="Full name" />

<!-- 3. Linked <label for=...> or wrapping <label> -->
<label for="fn">Full name</label><input id="fn" />

<!-- 4. title attribute (also provides tooltip) -->
<input title="Full name" />

<!-- 5. placeholder (LAST RESORT — disappears on input, avoid) -->
<input placeholder="Full name" />

<!-- If aria-labelledby and aria-label are both present,
     aria-labelledby wins. Always prefer visible labels
     (option 3) for the best user experience. -->

Understanding ARIA label computation priority helps debug "why is the screen reader saying that?". aria-labelledby always wins, then aria-label, then associated <label>, then title, then placeholder. Visible <label> elements are the gold standard for forms.
