Examples 5.3 — Keyboard Navigation

BASIC (1–13)

1. tabindex="0" — making a div keyboard focusable

<!-- WITHOUT tabindex: not reachable by Tab key -->
<div class="card">Product card</div>

<!-- WITH tabindex="0": enters the natural tab order -->
<div
  class="card"
  tabindex="0"
  role="button"
  aria-label="Widget A — $29.99, click to view details"
>
  Widget A — $29.99
</div>

<!-- tabindex="0" places the element in tab order
     at its natural DOM position. It does NOT jump
     ahead of other elements. -->

tabindex="0" adds an element to the page's tab order at its natural DOM position. Required whenever a non-interactive element (div, span, li) is given an interactive role like button, checkbox, or option. Always pair with an appropriate ARIA role.

2. tabindex="-1" — focusable by JS, not by Tab

<!-- Modal dialog: focus target, not in natural tab order -->
<div
  role="dialog"
  id="modal"
  aria-modal="true"
  aria-labelledby="modal-title"
  tabindex="-1"
>
  <h2 id="modal-title">Confirm action</h2>
  <p>Are you sure you want to continue?</p>
  <button>Confirm</button>
  <button>Cancel</button>
</div>

<script>
  function openModal() {
    const modal = document.getElementById('modal');
    modal.hidden = false;
    modal.focus(); // Programmatic focus via tabindex="-1"
  }
</script>

tabindex="-1" allows programmatic focus via JS .focus() but removes the element from the natural Tab sequence. Essential for dialog containers, headings that receive focus after navigation, and offscreen elements that should never be reached by Tab.

3. Positive tabindex — why to avoid it

<!-- BAD: positive tabindex disrupts natural tab order -->
<button tabindex="3">First in visual order</button>
<button tabindex="1">Second in visual order — but focused FIRST</button>
<button tabindex="2">Third in visual order — focused SECOND</button>

<!-- The Tab sequence becomes: tabindex=1, tabindex=2, tabindex=3,
     then all tabindex=0 elements, then DOM order for the rest.
     This creates a confusing, unpredictable order. -->

<!-- GOOD: Reorder the DOM instead -->
<button>Focused first (DOM order = visual order)</button>
<button>Focused second</button>
<button>Focused third</button>

Positive tabindex values create a separate tab order "pool" that is visited before any tabindex="0" elements, completely breaking the expected Tab sequence. The fix is always to reorder the DOM to match the visual reading order, not to patch it with positive values.

4. Skip navigation link

<!-- First element in body — visually hidden until focused -->
<a href="#main-content" class="skip-link">
  Skip to main content
</a>

<nav>
  <!-- 50+ navigation links... -->
</nav>

<main id="main-content" tabindex="-1">
  <h1>Page title</h1>
  <!-- Page content -->
</main>

<style>
  .skip-link {
    position: absolute;
    top: -40px;
    left: 0;
    background: #000;
    color: #fff;
    padding: 8px;
    text-decoration: none;
    z-index: 100;
  }
  .skip-link:focus {
    top: 0; /* Revealed on focus */
  }
</style>

Skip links allow keyboard users to bypass repetitive navigation and jump directly to main content. The link is visually hidden off-screen until focused, then slides into view. The main element needs tabindex="-1" so it can receive programmatic focus from the anchor.

5. :focus styles — never suppress the focus ring

<style>
  /* BAD: removes focus visibility entirely */
  *:focus { outline: none; }
  button:focus { outline: 0; }

  /* GOOD: customise the focus ring with a visible style */
  :focus-visible {
    outline: 3px solid #005fcc;
    outline-offset: 2px;
    border-radius: 2px;
  }

  /* GOOD: enhanced focus ring for interactive elements */
  button:focus-visible,
  a:focus-visible,
  [tabindex]:focus-visible {
    outline: 3px solid #005fcc;
    outline-offset: 3px;
    box-shadow: 0 0 0 5px rgba(0, 95, 204, 0.3);
  }
</style>

Removing focus outlines is one of the most harmful accessibility mistakes. 2.6% of users navigate by keyboard only. Use :focus-visible to show a clear ring only for keyboard navigation (not mouse clicks). WCAG 2.4.11 requires 3:1 contrast ratio for focus indicators.

6. :focus-visible — keyboard-only focus indicator

<style>
  /* Applies only when the element was focused via keyboard or AT,
     not when focused by mouse click */
  button:focus-visible {
    outline: 3px solid #0078d7;
    outline-offset: 2px;
  }

  /* Remove outline for mouse users only (still shown for keyboard) */
  button:focus:not(:focus-visible) {
    outline: none;
  }
</style>

<!-- :focus-visible is supported in all modern browsers.
     For older browsers: use the focus-visible polyfill
     (npm install focus-visible) which adds a .focus-visible class -->

:focus-visible solves the UX problem of unwanted focus rings on mouse clicks while preserving them for keyboard navigation. Browsers apply it using heuristics: keyboard Tab always triggers it; mouse clicks on buttons typically don't.

7. accesskey attribute

<!-- Provides a keyboard shortcut to focus/activate the element -->
<a href="/search" accesskey="s">
  <u>S</u>earch
</a>

<button accesskey="n">
  <u>N</u>ew document
</button>

<!-- Activation shortcuts vary by OS + browser:
     Windows/Linux: Alt + key (Chrome/Firefox)
     Mac:          Ctrl + Option + key (Chrome)
                   Ctrl + Alt    + key (Firefox)

     Caveats:
     - May conflict with browser/OS shortcuts
     - Not predictable across platforms
     - Rarely used correctly — document all shortcuts
-->

accesskey provides platform-specific keyboard shortcuts. Underline the access key letter for discoverability (u element, not text-decoration:underline). Use sparingly — conflicts with screen reader shortcut keys are common. Document all shortcuts in a visible keyboard reference.

8. Natural Tab order follows DOM order

<!-- Tab visits elements in DOM source order -->
<header>
  <a href="/">Home</a>          <!-- Tab 1 -->
  <a href="/about">About</a>    <!-- Tab 2 -->
</header>
<main>
  <h1>Products</h1>
  <button>Filter</button>       <!-- Tab 3 -->
  <a href="/item/1">Widget</a>  <!-- Tab 4 -->
  <a href="/item/2">Gadget</a>  <!-- Tab 5 -->
</main>
<footer>
  <a href="/contact">Contact</a><!-- Tab 6 -->
</footer>

<!-- CSS order / transform / float do NOT affect Tab order.
     Only DOM source order matters. -->

The browser's Tab sequence follows DOM source order, regardless of visual positioning via CSS. Layouts achieved with CSS Grid or Flexbox order:N can create a mismatch between visual and keyboard order — a WCAG 1.3.2 / 2.4.3 violation. Keep DOM order meaningful.

9. Shift+Tab — reverse navigation

<!-- Shift+Tab moves focus to the previous focusable element -->
<!-- The order is the reverse of the Tab sequence (DOM order) -->

<form>
  <label for="first">First name</label>
  <input id="first" type="text">     <!-- Tab 1 / Shift+Tab 4 -->

  <label for="last">Last name</label>
  <input id="last" type="text">      <!-- Tab 2 / Shift+Tab 3 -->

  <button type="button">Cancel</button> <!-- Tab 3 / Shift+Tab 2 -->
  <button type="submit">Save</button>   <!-- Tab 4 / Shift+Tab 1 -->
</form>

<!-- Focus trap (modal): both Tab and Shift+Tab must remain
     inside the dialog, wrapping around at boundaries -->

Shift+Tab is the reverse Tab. Keyboard users rely on it to correct mistakes by moving backward through forms. Focus traps must intercept both Tab and Shift+Tab to keep focus within a modal dialog, preventing escape to the page behind.

10. Enter and Space on buttons

<!-- Native button: Enter AND Space both activate it -->
<button onclick="submit()">Submit</button>

<!-- Native link: Enter activates it, Space scrolls page -->
<a href="/next">Next</a>

<!-- Custom button (div): you MUST add both -->
<div
  role="button"
  tabindex="0"
  id="custom"
>Custom</div>

<script>
  document.getElementById('custom').addEventListener('keydown', e => {
    if (e.key === 'Enter') { e.preventDefault(); activate(); }
    if (e.key === ' ')     { e.preventDefault(); activate(); }
    // Space preventDefault stops page scroll
  });
</script>

Native <button> fires click on both Enter and Space. Native <a> fires on Enter only. Custom role="button" elements must manually handle both keys. Forgetting Space is the most common keyboard bug in custom button implementations.

11. Focus visible check — testing keyboard navigation

<!-- Keyboard navigation testing checklist:
  1. Press Tab — every interactive element should receive visible focus
  2. Press Shift+Tab — reverse order works correctly
  3. No focus trap on the main page (Tab should never get stuck)
  4. Modals/dropdowns trap focus correctly
  5. Escape closes modal/dropdown and returns focus to trigger
  6. Arrow keys work inside composite widgets
  7. Enter/Space activate buttons and links
  8. Form submit works with Enter in text fields
-->

<!-- Automated check: find all focusable elements -->
<script>
  const focusable = document.querySelectorAll(
    'a[href], button:not([disabled]), input:not([disabled]), ' +
    'select:not([disabled]), textarea:not([disabled]), ' +
    '[tabindex]:not([tabindex="-1"]), details > summary'
  );
  console.log('Focusable elements:', focusable.length, focusable);
</script>

Manual keyboard testing is the most reliable method. Tab through the entire page and verify every interactive element is reachable and has a visible focus indicator. The JS snippet logs all focusable elements for auditing.

12. Enter in a text input submits the form

<form id="search-form">
  <label for="q">Search</label>
  <input type="search" id="q" name="q">
  <!-- Enter key in the input submits this form natively -->
  <button type="submit">Search</button>
</form>

<!-- Prevent default submit while still allowing Enter: -->
<script>
  document.getElementById('search-form').addEventListener('submit', e => {
    e.preventDefault();
    const query = document.getElementById('q').value;
    performSearch(query);
  });
  // Do NOT intercept keydown on the input to prevent Enter submit —
  // that breaks expected behavior for keyboard and AT users.
</script>

Pressing Enter while focus is in a text input submits the containing form — a universal browser convention. Never suppress this with a keydown listener on the input. Instead, handle the form's submit event, which fires for both Enter-key and button-click submissions.

13. tabindex on non-interactive elements — use sparingly

<!-- Only make elements focusable if they are interactive -->

<!-- BAD: paragraph in tab order serves no purpose -->
<p tabindex="0">This is some information.</p>

<!-- GOOD: heading as navigation target (tabindex="-1") -->
<h2 tabindex="-1" id="results-heading">Search results</h2>
<!-- Focused programmatically after search, not by Tab -->

<!-- GOOD: scrollable region with keyboard-accessible scroll -->
<div
  tabindex="0"
  role="region"
  aria-label="Terms and conditions"
  style="overflow-y: auto; max-height: 200px"
>
  <!-- Long text... -->
</div>

Add tabindex="0" only when: (a) the element has an interactive role, or (b) it is a scrollable region that must be keyboard-navigable. Content-only elements (paragraphs, headings) belong in the tab order only when they serve as programmatic focus targets (use tabindex="-1").

INTERMEDIATE (14–26)

14. Focus management in SPA routing

<!-- Persistent route announcer -->
<div role="status" id="route-announcer" class="sr-only"></div>

<main id="main-content" tabindex="-1">
  <!-- Route content rendered here -->
</main>

<script>
  // Called on every route change
  function handleRouteChange(newPath, pageTitle) {
    const main      = document.getElementById('main-content');
    const announcer = document.getElementById('route-announcer');

    // 1. Render new page content
    renderPage(newPath);

    // 2. Announce navigation
    announcer.textContent = '';
    setTimeout(() => {
      announcer.textContent = `Navigated to ${pageTitle}`;
    }, 100);

    // 3. Move focus to main (users can then Tab into content)
    main.focus();
  }
</script>

SPA navigation must move focus and announce the new page. Without this, keyboard users stay focused on the clicked link (now on the old page). The sequence: render, announce (delayed), then focus main. This restores the experience of a full page load.

15. Focus trap in a modal dialog

<div role="dialog" id="modal" aria-modal="true" aria-labelledby="modal-h" hidden>
  <h2 id="modal-h">Delete account</h2>
  <p>This action is permanent.</p>
  <button id="modal-cancel">Cancel</button>
  <button id="modal-confirm">Delete</button>
</div>

<script>
  const FOCUSABLE = 'button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])';

  function trapFocus(modal) {
    const focusableEls = [...modal.querySelectorAll(FOCUSABLE)];
    const first = focusableEls[0];
    const last  = focusableEls[focusableEls.length - 1];

    modal.addEventListener('keydown', function handler(e) {
      if (e.key !== 'Tab') return;
      if (e.shiftKey) {
        if (document.activeElement === first) {
          e.preventDefault(); last.focus();
        }
      } else {
        if (document.activeElement === last) {
          e.preventDefault(); first.focus();
        }
      }
    });
  }

  function openModal() {
    const modal = document.getElementById('modal');
    modal.hidden = false;
    trapFocus(modal);
    document.getElementById('modal-cancel').focus();
  }
</script>

A focus trap intercepts Tab and Shift+Tab at the boundary elements, wrapping around instead of exiting. Focus must be moved inside on open and returned to the trigger on close. This is a WCAG 2.1.2 requirement — users must be able to navigate to and from every component.

16. Roving tabindex pattern

<div role="toolbar" aria-label="Text formatting">
  <button tabindex="0"  id="tb-bold"   aria-pressed="false">Bold</button>
  <button tabindex="-1" id="tb-italic" aria-pressed="false">Italic</button>
  <button tabindex="-1" id="tb-under"  aria-pressed="false">Underline</button>
</div>

<script>
  const toolbar = document.querySelector('[role="toolbar"]');
  const buttons = [...toolbar.querySelectorAll('button')];

  toolbar.addEventListener('keydown', e => {
    const idx = buttons.indexOf(document.activeElement);
    let next = idx;

    if (e.key === 'ArrowRight') { e.preventDefault(); next = (idx + 1) % buttons.length; }
    if (e.key === 'ArrowLeft')  { e.preventDefault(); next = (idx - 1 + buttons.length) % buttons.length; }
    if (e.key === 'Home')       { e.preventDefault(); next = 0; }
    if (e.key === 'End')        { e.preventDefault(); next = buttons.length - 1; }

    if (next !== idx) {
      buttons[idx].tabIndex = -1;  // Remove previous from tab order
      buttons[next].tabIndex = 0;  // Add new to tab order
      buttons[next].focus();
    }
  });
</script>

Roving tabindex keeps exactly one tab stop in a composite widget. Only the active item has tabindex="0"; others have tabindex="-1". Arrow keys move focus between items, updating tabindex accordingly. Tab moves focus out of the widget entirely.

17. Arrow key patterns for composite widgets

<!-- Pattern: single Tab stop into widget, then arrow keys inside -->

<!-- Horizontal widget: Left/Right arrows -->
<div role="tablist"> <!-- Left/Right arrows to switch tabs --></div>

<!-- Vertical widget: Up/Down arrows -->
<ul role="listbox"> <!-- Up/Down arrows to move options --></ul>

<!-- Two-dimensional: all four arrow keys -->
<table role="grid"> <!-- Arrow keys to move between cells --></table>

<!-- Tree: Up/Down to move, Right to expand/move into, Left to collapse -->
<ul role="tree">...</ul>

<!-- Home: move to first item  / Ctrl+Home: first item in widget -->
<!-- End:  move to last item   / Ctrl+End:  last item in widget  -->
<!-- Enter/Space: activate selected item                         -->
<!-- Escape: close popup / cancel operation                      -->
<!-- Type-ahead: first character to jump to matching item        -->

Composite widgets use internal arrow-key navigation to avoid polluting the page's Tab sequence. Each widget type has a specific keyboard pattern defined by ARIA Authoring Practices Guide (APG). Implementing these correctly provides a native-app experience.

18. Escape key handler

<script>
  // Global Escape key handler
  document.addEventListener('keydown', e => {
    if (e.key !== 'Escape') return;

    // Close modal if open
    const modal = document.querySelector('[role="dialog"]:not([hidden])');
    if (modal) { closeModal(modal); return; }

    // Close dropdown if open
    const dropdown = document.querySelector('[aria-expanded="true"]');
    if (dropdown) { closeDropdown(dropdown); return; }

    // Close tooltip if showing
    const tooltip = document.querySelector('[role="tooltip"]:not([hidden])');
    if (tooltip) { closeTooltip(tooltip); return; }
  });

  function closeModal(modal) {
    modal.hidden = true;
    // Return focus to the element that opened it
    const trigger = document.getElementById(modal.dataset.triggerId);
    trigger?.focus();
  }
</script>

Escape is the universal "exit" key for popups, dialogs, menus, and tooltips (WCAG 1.4.13 for non-hover content). Handle it globally and work outward from the most recently opened layer. Always return focus to the trigger element on close.

19. Focus return after modal close

<button id="open-modal-btn" onclick="openModal(this)">
  Edit profile
</button>

<div role="dialog" id="edit-modal" aria-modal="true" hidden
     data-trigger-id="open-modal-btn">
  <h2>Edit profile</h2>
  <button onclick="closeModal('edit-modal')">Cancel</button>
  <button onclick="saveAndClose('edit-modal')">Save</button>
</div>

<script>
  let returnFocusEl = null;

  function openModal(trigger) {
    returnFocusEl = trigger; // Save trigger reference
    const modal = document.getElementById('edit-modal');
    modal.hidden = false;
    modal.querySelector('button').focus(); // Focus first button
  }

  function closeModal(id) {
    document.getElementById(id).hidden = true;
    returnFocusEl?.focus(); // Return focus to trigger
    returnFocusEl = null;
  }
</script>

Storing the trigger element before opening a modal and restoring focus to it on close is essential for orientation. Without this, keyboard users lose their place in the page after the modal closes — they would have to Tab from the beginning to find their context again.

20. Keyboard shortcuts with modifier keys

<script>
  document.addEventListener('keydown', e => {
    // Ignore shortcuts when typing in inputs
    const tag = document.activeElement.tagName;
    if (['INPUT', 'TEXTAREA', 'SELECT'].includes(tag)) return;
    if (document.activeElement.isContentEditable) return;

    // Ctrl/Cmd + S = Save
    if ((e.ctrlKey || e.metaKey) && e.key === 's') {
      e.preventDefault();
      saveDocument();
    }

    // Ctrl/Cmd + Z = Undo
    if ((e.ctrlKey || e.metaKey) && !e.shiftKey && e.key === 'z') {
      e.preventDefault();
      undoAction();
    }

    // Ctrl/Cmd + Shift + Z = Redo
    if ((e.ctrlKey || e.metaKey) && e.shiftKey && e.key === 'z') {
      e.preventDefault();
      redoAction();
    }
  });
</script>

<!-- Document shortcuts in a visible keyboard help panel -->
<button onclick="showKeyboardHelp()">Keyboard shortcuts (?)</button>

Custom keyboard shortcuts should: use modifier keys to avoid conflicts with AT shortcuts; ignore shortcuts when focus is in an input; mirror familiar conventions (Ctrl+S=save, Ctrl+Z=undo); and be discoverable via a documented keyboard shortcut help panel.

21. Skip nav for complex pages — multiple skip targets

<!-- Multiple skip targets for complex page structures -->
<div class="skip-links">
  <a href="#main-content"  class="skip-link">Skip to main content</a>
  <a href="#site-search"   class="skip-link">Skip to search</a>
  <a href="#site-nav"      class="skip-link">Skip to navigation</a>
</div>

<header>
  <form id="site-search" role="search" aria-label="Site search">
    <!-- search -->
  </form>
  <nav id="site-nav" aria-label="Main">
    <!-- navigation -->
  </nav>
</header>

<main id="main-content" tabindex="-1">
  <!-- content -->
</main>

<style>
  .skip-links { position: absolute; }
  .skip-link { position: absolute; top: -60px; }
  .skip-link:focus { top: 0; }
</style>

Pages with large headers benefit from multiple skip targets. All skip links are revealed on focus and hidden via absolute positioning. Each target uses tabindex="-1" to accept programmatic focus from the anchor without appearing in the natural Tab sequence.

22. Type-ahead in a listbox

<ul
  role="listbox"
  id="country-list"
  tabindex="0"
  aria-label="Select country"
  aria-activedescendant=""
>
  <li role="option" id="opt-au">Australia</li>
  <li role="option" id="opt-br">Brazil</li>
  <li role="option" id="opt-ca">Canada</li>
  <!-- ... 200 more countries -->
</ul>

<script>
  let typeaheadBuffer = '';
  let typeaheadTimer;

  document.getElementById('country-list').addEventListener('keydown', e => {
    if (e.key.length !== 1) return; // Ignore non-printable keys

    typeaheadBuffer += e.key.toLowerCase();
    clearTimeout(typeaheadTimer);
    typeaheadTimer = setTimeout(() => { typeaheadBuffer = ''; }, 500);

    const options = [...document.querySelectorAll('[role="option"]')];
    const match = options.find(o =>
      o.textContent.trim().toLowerCase().startsWith(typeaheadBuffer)
    );
    if (match) {
      document.getElementById('country-list')
        .setAttribute('aria-activedescendant', match.id);
      match.scrollIntoView({ block: 'nearest' });
    }
  });
</script>

Type-ahead lets users jump to an option by typing its first character(s). A 500ms buffer allows multi-character prefix matching ("Ca" → Canada). This matches native <select> behavior that keyboard users rely on for long lists.

23. Focus indicator — high-contrast mode

<style>
  /* Standard focus ring */
  :focus-visible {
    outline: 3px solid #0058cc;
    outline-offset: 2px;
  }

  /* Windows High Contrast Mode (forced-colors) */
  @media (forced-colors: active) {
    :focus-visible {
      /* Use system highlight color — don't override with a fixed color */
      outline: 3px solid Highlight;
      outline-offset: 2px;
    }
  }

  /* Ensure focus ring is visible on dark AND light backgrounds */
  :focus-visible {
    outline: 3px solid #0058cc;
    outline-offset: 2px;
    /* Add white ring inside for dark backgrounds */
    box-shadow:
      0 0 0 1px #fff,
      0 0 0 4px #0058cc;
  }
</style>

Focus indicators must be visible in high-contrast mode. Use forced-colors media query and system color keywords (Highlight) rather than fixed colors. The double-ring technique (white then colored) ensures visibility on both dark and light backgrounds.

24. Programmatic focus with announcements

<div id="wizard">
  <div id="step-1">
    <h2 tabindex="-1" id="step-1-h">Step 1: Personal details</h2>
    <!-- form fields -->
    <button onclick="goToStep(2)">Continue</button>
  </div>

  <div id="step-2" hidden>
    <h2 tabindex="-1" id="step-2-h">Step 2: Address</h2>
    <!-- form fields -->
    <button onclick="goToStep(3)">Continue</button>
  </div>
</div>

<div role="status" id="wizard-status"></div>

<script>
  function goToStep(n) {
    // Hide current step
    document.getElementById(`step-${n-1}`).hidden = true;
    // Show new step
    const newStep = document.getElementById(`step-${n}`);
    newStep.hidden = false;

    // Announce then focus
    document.getElementById('wizard-status').textContent =
      `Step ${n} of 4`;
    setTimeout(() => {
      document.getElementById(`step-${n}-h`).focus();
    }, 100);
  }
</script>

When displaying new content sections (wizard steps, loaded panels), announce the change via a status region then move focus to the section heading after a small delay. The delay ensures the announcement is not interrupted by the focus event.

25. WCAG 2.1 keyboard requirements

<!-- WCAG 2.1 keyboard-related success criteria:

  1.3.1  Info & Relationships (A) — Structure conveyed programmatically
  1.3.2  Meaningful Sequence (A) — DOM order = logical reading order
  1.4.11 Non-text Contrast (AA) — Focus indicator: 3:1 contrast
  1.4.13 Content on Hover/Focus (AA) — Tooltips dismissible + hoverable
  2.1.1  Keyboard (A) — ALL functionality via keyboard (no mouse-only paths)
  2.1.2  No Keyboard Trap (A) — Users can always navigate away
  2.1.4  Character Key Shortcuts (A) — Single-key shortcuts can be disabled
  2.4.3  Focus Order (A) — Focus sequence is logical
  2.4.7  Focus Visible (AA) — Keyboard focus always visible
  2.4.11 Focus Appearance (AAA) — Focus indicator size and contrast
  2.4.12 Focus Not Obscured (AA, WCAG 2.2) — Focus ring not hidden by sticky headers
  3.2.1  On Focus (A) — Focus alone does not change context

-->

These WCAG criteria form the keyboard accessibility compliance checklist. 2.1.1 (keyboard access) and 2.1.2 (no keyboard trap) are Level A requirements — the absolute minimum. 2.4.7 (focus visible) is Level AA, required for most public-sector and large-enterprise applications.

26. Focus not obscured by sticky header (WCAG 2.4.12)

<style>
  /* Sticky header that could hide focused elements */
  header {
    position: sticky;
    top: 0;
    height: 60px;
    background: white;
    z-index: 100;
  }

  /* Fix 1: scroll-padding-top ensures focused elements appear below header */
  html {
    scroll-padding-top: 70px; /* header height + buffer */
  }

  /* Fix 2: scroll-margin-top on interactive elements */
  a:focus,
  button:focus,
  input:focus,
  [tabindex]:focus {
    scroll-margin-top: 70px;
  }
</style>

WCAG 2.4.12 (WCAG 2.2, Level AA) requires that focused elements are not entirely obscured by sticky/fixed UI. scroll-padding-top on the html element is the simplest fix — browsers automatically scroll focused elements below the sticky header amount.

NESTED (27–38)

27. Modal dialog focus trap — complete implementation

<button id="settings-trigger">Open settings</button>

<div
  id="settings-dialog"
  role="dialog"
  aria-modal="true"
  aria-labelledby="settings-title"
  hidden
>
  <h2 id="settings-title" tabindex="-1">Settings</h2>
  <label><input type="checkbox"> Dark mode</label>
  <label><input type="checkbox"> Notifications</label>
  <button id="settings-save">Save</button>
  <button id="settings-close">Close</button>
</div>

<script>
  const FOCUSABLE_SEL =
    'a[href], button:not([disabled]), input, select, textarea, [tabindex]:not([tabindex="-1"])';

  let returnTo = null;

  function openDialog() {
    returnTo = document.activeElement;
    const dlg = document.getElementById('settings-dialog');
    dlg.hidden = false;
    dlg.querySelector(FOCUSABLE_SEL).focus();
    dlg.addEventListener('keydown', trapHandler);
  }

  function closeDialog() {
    const dlg = document.getElementById('settings-dialog');
    dlg.hidden = true;
    dlg.removeEventListener('keydown', trapHandler);
    returnTo?.focus();
    returnTo = null;
  }

  function trapHandler(e) {
    const dlg = document.getElementById('settings-dialog');
    const els = [...dlg.querySelectorAll(FOCUSABLE_SEL)];
    const first = els[0], last = els[els.length - 1];

    if (e.key === 'Escape') { e.preventDefault(); closeDialog(); return; }
    if (e.key !== 'Tab') return;

    if (e.shiftKey && document.activeElement === first) {
      e.preventDefault(); last.focus();
    } else if (!e.shiftKey && document.activeElement === last) {
      e.preventDefault(); first.focus();
    }
  }

  document.getElementById('settings-trigger').addEventListener('click', openDialog);
  document.getElementById('settings-close').addEventListener('click', closeDialog);
</script>

Complete modal focus trap: save trigger reference, move focus to first focusable element on open, intercept Tab/Shift+Tab to wrap within the dialog, handle Escape to close, and return focus to the trigger. Remove the keydown listener on close to avoid memory leaks.

29. Tab widget keyboard pattern

<div role="tablist" aria-label="Account">
  <button role="tab" id="t1" aria-selected="true"  aria-controls="p1" tabindex="0">Profile</button>
  <button role="tab" id="t2" aria-selected="false" aria-controls="p2" tabindex="-1">Security</button>
  <button role="tab" id="t3" aria-selected="false" aria-controls="p3" tabindex="-1">Billing</button>
</div>

<script>
  const tabs = [...document.querySelectorAll('[role="tab"]')];

  document.querySelector('[role="tablist"]').addEventListener('keydown', e => {
    const idx = tabs.indexOf(document.activeElement);
    let next = idx;

    if (e.key === 'ArrowRight') next = (idx + 1) % tabs.length;
    if (e.key === 'ArrowLeft')  next = (idx - 1 + tabs.length) % tabs.length;
    if (e.key === 'Home')       next = 0;
    if (e.key === 'End')        next = tabs.length - 1;

    if (next !== idx) {
      e.preventDefault();
      activateTab(tabs[next]);
    }
  });

  function activateTab(tab) {
    tabs.forEach(t => {
      t.setAttribute('aria-selected', 'false');
      t.tabIndex = -1;
      document.getElementById(t.getAttribute('aria-controls')).hidden = true;
    });
    tab.setAttribute('aria-selected', 'true');
    tab.tabIndex = 0;
    tab.focus();
    document.getElementById(tab.getAttribute('aria-controls')).hidden = false;
  }
</script>

Tab widget keyboard: Left/Right arrows move between tabs using roving tabindex; Home/End go to first/last. This example uses "automatic activation" (selecting on arrow key). An alternative is "manual activation" (arrow moves focus, Enter/Space selects) — useful when panels are expensive to load.

    draggable="true"
    tabindex="0"
    aria-describedby="dnd-help"
    data-index="0"
  >Item 1</li>
  <li draggable="true" tabindex="0" aria-describedby="dnd-help" data-index="1">Item 2</li>
  <li draggable="true" tabindex="0" aria-describedby="dnd-help" data-index="2">Item 3</li>

<script>
  let pickedUp = null;

  document.getElementById('sortable-list').addEventListener('keydown', e => {
    const item = e.target.closest('li');

      e.preventDefault();

        item.setAttribute('aria-grabbed', 'true');

35. Nested focus trap — stacked dialogs

<div id="dialog-1" role="dialog" aria-modal="true" aria-labelledby="d1-title" hidden>
  <h2 id="d1-title">Edit order</h2>
  <button onclick="openDialog2()">Change shipping</button>
  <button onclick="closeDialog1()">Close</button>
</div>

<div id="dialog-2" role="dialog" aria-modal="true" aria-labelledby="d2-title" hidden>
  <h2 id="d2-title">Select shipping method</h2>
  <button onclick="selectStandard()">Standard</button>
  <button onclick="closeDialog2()">Cancel</button>
</div>

<script>
  const trapStack = [];

  function openDialog2() {
    // Save current trap context and install new one
    trapStack.push(document.getElementById('dialog-1'));
    installTrap(document.getElementById('dialog-2'));
    document.getElementById('dialog-2').hidden = false;
    document.getElementById('dialog-2').querySelector('button').focus();

  function closeDialog2() {
    document.getElementById('dialog-2').hidden = true;
    removeTrap();
    // Return focus to dialog-1, not to the original page trigger
    const prev = trapStack.pop();
    prev?.querySelector('button')?.focus();
  }
</script>

Stacked dialogs require a trap stack. When a second dialog opens, push the first dialog context onto the stack. On close, pop it and return focus to the appropriate element within the previous layer. The first dialog's trap resumes naturally.

39. Virtual focus with aria-activedescendant

<!-- Container holds real DOM focus; activeDescendant tracks virtual focus -->
<div
  id="autocomplete-list"
  role="listbox"
  tabindex="0"
  aria-label="Search suggestions"
  aria-activedescendant=""
>
  <div role="option" id="s-opt-0">Accessibility</div>
  <div role="option" id="s-opt-1">Accessible design</div>
  <div role="option" id="s-opt-2">ARIA authoring</div>
</div>

<script>
  let virtualIdx = -1;
  const list = document.getElementById('autocomplete-list');
  const opts = () => [...list.querySelectorAll('[role="option"]')];

  list.addEventListener('keydown', e => {
    const options = opts();
    if (e.key === 'ArrowDown') {
      e.preventDefault();
      virtualIdx = (virtualIdx + 1) % options.length;
    } else if (e.key === 'ArrowUp') {
      e.preventDefault();
      virtualIdx = (virtualIdx - 1 + options.length) % options.length;
    }

    options.forEach((o, i) => {
      o.classList.toggle('highlighted', i === virtualIdx);
    });

    list.setAttribute('aria-activedescendant',
      virtualIdx >= 0 ? options[virtualIdx].id : ''
    );
  });
</script>

Virtual focus keeps one real DOM focus point (the container) while ARIA tracks a virtual focus position via aria-activedescendant. Screen readers read the active descendant, not the container. This avoids janky focus movement in large lists and allows smooth keyboard traversal.

48. Reading order vs DOM order — detecting mismatches

<!-- Visual order (CSS Grid) mismatches DOM order -->
<style>
  .grid { display: grid; grid-template-areas: 'sidebar main'; }
  .sidebar { grid-area: sidebar; }
  .main    { grid-area: main; }
</style>

<div class="grid">
  <main class="main">
    <a href="/item1">Item 1</a> <!-- DOM first, visual right -->
  </main>
  <aside class="sidebar">
    <a href="/filter">Filter</a> <!-- DOM second, visual left -->
  </aside>
</div>
<!-- Tab order: Item 1 first, Filter second — WRONG -->

<!-- Fix: reorder DOM to match visual order -->
<div class="grid">
  <aside class="sidebar">
    <a href="/filter">Filter</a> <!-- DOM first, visual left -->
  </aside>
  <main class="main">
    <a href="/item1">Item 1</a> <!-- DOM second, visual right -->
  </main>
</div>
<!-- Tab order: Filter first, Item 1 second — CORRECT -->

CSS Grid and Flexbox order properties change visual order without changing DOM order. Tab follows DOM order. When a sidebar is visually first but DOM-second, keyboard users encounter content in the wrong sequence — a WCAG 1.3.2 (Meaningful Sequence) and 2.4.3 (Focus Order) violation.

49. inert attribute for blocking keyboard access

<!-- inert: prevents focus, pointer events, and AT discovery -->

<!-- Background content when modal is open -->
<div id="page-content" inert aria-hidden="true">
  <nav>...</nav>
  <main>...</main>
</div>

<!-- Active modal -->
<div role="dialog" aria-modal="true">
  <!-- Only content here is keyboard-accessible -->
</div>

<script>
  function openModal() {
    document.getElementById('page-content').inert = true;
    document.getElementById('page-content').setAttribute('aria-hidden', 'true');
    document.getElementById('modal').hidden = false;
    document.getElementById('modal').querySelector('[autofocus]')?.focus();
  }

  function closeModal() {
    document.getElementById('modal').hidden = true;
    document.getElementById('page-content').removeAttribute('inert');
    document.getElementById('page-content').removeAttribute('aria-hidden');
    returnFocusEl?.focus();
  }
</script>

The inert attribute is the modern replacement for manual focus trap implementations. When a modal is open, set the entire page content inert — this blocks Tab, keyboard interaction, pointer events, and removes everything from the AT tree automatically. Supported in all major browsers since 2023.
