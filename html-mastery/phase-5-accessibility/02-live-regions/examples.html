Examples 5.2 — Live Regions

BASIC (1–13)

1. aria-live="polite" — the safe default

<div aria-live="polite" id="notification-area">
  <!-- Content inserted here will be announced
       after the user finishes their current activity -->
</div>

<script>
  function notify(message) {
    document.getElementById('notification-area').textContent = message;
  }

  // Called after a successful action:
  notify('Profile updated successfully.');
</script>

aria-live="polite" queues announcements until the user pauses. It is the correct choice for non-urgent updates like save confirmations, filter results, and cart changes. The region must be in the DOM before content is injected.

2. aria-live="assertive" — interrupt immediately

<div aria-live="assertive" id="error-region">
  <!-- Content here interrupts the screen reader
       immediately, even mid-sentence -->
</div>

<script>
  function announceError(msg) {
    document.getElementById('error-region').textContent = msg;
  }

  // Called on critical session timeout:
  announceError('Session expired. Please log in again.');
</script>

aria-live="assertive" interrupts any current AT reading to announce the new content immediately. Use only for critical errors, session timeouts, or situations where delayed announcement causes real harm. Overuse makes the experience extremely jarring.

3. aria-live="off" — suppress announcements

<!-- The region updates visually but AT ignores the changes -->
<div
  aria-live="off"
  id="visual-ticker"
  aria-label="Stock prices (audio notifications disabled)"
>
  AAPL $192.34 ▲ | GOOG $178.50 ▼ | MSFT $420.10 ▲
</div>

<!-- User controls whether to hear these updates: -->
<button onclick="toggleAudio()">Enable audio updates</button>

<script>
  let audioOn = false;
  function toggleAudio() {
    audioOn = !audioOn;
    const ticker = document.getElementById('visual-ticker');
    ticker.setAttribute('aria-live', audioOn ? 'polite' : 'off');
  }
</script>

aria-live="off" (the default for most elements) suppresses announcements. Switching the attribute value dynamically lets users opt in to audio updates for high-frequency data feeds without constant interruption.

4. role="status" — polite status messages

<!-- role="status" implies: aria-live="polite" + aria-atomic="true" -->
<div role="status" id="form-status"></div>

<form>
  <button type="submit">Save changes</button>
</form>

<script>
  document.querySelector('form').addEventListener('submit', async (e) => {
    e.preventDefault();
    await saveData();
    document.getElementById('form-status').textContent =
      'Changes saved successfully at ' + new Date().toLocaleTimeString();
  });
</script>

role="status" carries implicit aria-live="polite" and aria-atomic="true". It is the go-to container for save confirmations, status messages, and non-critical updates. Place it in the DOM on page load — never inject the element itself dynamically.

5. role="alert" — urgent error announcements

<!-- role="alert" implies: aria-live="assertive" + aria-atomic="true" -->
<div role="alert" id="form-alert" aria-label="Form error">
  <!-- Empty on load; error injected on validation failure -->
</div>

<script>
  function showError(message) {
    const alert = document.getElementById('form-alert');
    // Clear first, then set — ensures re-announcement of same message
    alert.textContent = '';
    requestAnimationFrame(() => {
      alert.textContent = message;
    });
  }

  showError('Invalid credit card number. Please check and try again.');
</script>

role="alert" forces immediate announcement. The clear-then-set pattern with requestAnimationFrame ensures the screen reader treats it as a new mutation even if the text is the same as before. Use sparingly — only for genuine errors.

6. role="log" — for chat and event streams

<!-- role="log" implies aria-live="polite" -->
<!-- New items are announced as they are appended -->
<div
  role="log"
  aria-label="Chat messages"
  aria-relevant="additions"
  id="chat-log"
>
  <div><strong>Alice:</strong> Hello!</div>
  <div><strong>Bob:</strong> Hi there!</div>
</div>

<script>
  function addMessage(author, text) {
    const log = document.getElementById('chat-log');
    const msg = document.createElement('div');
    msg.innerHTML = `<strong>${author}:</strong> ${text}`;
    log.appendChild(msg); // Appended content is announced
  }
</script>

role="log" is designed for chronological logs where only additions matter. It implies polite live updates. Unlike role="alert", the log accumulates content and AT announces new additions without repeating old messages.

7. Adding content to a live region correctly

<div role="status" id="status"></div>

<script>
  // CORRECT: Modify the live region's content after it's in the DOM
  document.getElementById('status').textContent = 'Item added to cart.';

  // CORRECT: Append a child node
  const p = document.createElement('p');
  p.textContent = '3 items in cart.';
  document.getElementById('status').appendChild(p);

  // WRONG: Inject the live region element itself into the DOM
  // (AT may miss the mutation when the element appears)
  const bad = document.createElement('div');
  bad.setAttribute('role', 'status');
  bad.textContent = 'Item added to cart.';
  document.body.appendChild(bad); // <-- AT often ignores this
</script>

The live region element must exist in the DOM when the page loads (even if empty). Screen readers register live regions during page parse. Dynamically injecting the live region element itself frequently causes announcements to be silently dropped.

8. Clearing a live region

<div role="status" id="msg"></div>
<button onclick="clearMsg()">Dismiss</button>

<script>
  function clearMsg() {
    // Simply empty the live region — no announcement is made for removal
    document.getElementById('msg').textContent = '';
    // Alternatively: innerHTML = ''
  }

  // Pattern: always clear before setting, then set in next frame
  function announce(text) {
    const el = document.getElementById('msg');
    el.textContent = '';
    requestAnimationFrame(() => { el.textContent = text; });
  }
</script>

Removing content from a live region does not trigger an announcement (by default; see aria-relevant). Clearing the region silently hides the message. The rAF pattern reliably re-announces the same text by creating two distinct DOM mutations.

9. aria-atomic="true" — announce the whole region

<!-- With aria-atomic="true", the entire region is announced
     as one unit when any part changes -->
<div
  aria-live="polite"
  aria-atomic="true"
  id="clock"
>
  <span id="time">12:00</span>
  <span id="ampm">PM</span>
</div>

<script>
  setInterval(() => {
    const now = new Date();
    document.getElementById('time').textContent =
      now.toLocaleTimeString('en', { hour: '2-digit', minute: '2-digit' });
    // Even though only #time changes, AT reads the whole div:
    // "12:34 PM"
  }, 60000);
</script>

aria-atomic="true" ensures AT reads the entire container when any descendant changes, not just the changed portion. Essential when partial content is meaningless in isolation (e.g., time "12:34" needs the "PM" part to be complete).

10. aria-atomic="false" — announce only the change

<!-- With aria-atomic="false", only the changed node is announced -->
<div
  aria-live="polite"
  aria-atomic="false"
  id="score-board"
>
  <p>Player: Alice</p>
  <p id="score">Score: 0</p>
  <p>Level: 3</p>
</div>

<script>
  function updateScore(points) {
    // Only "Score: 150" is announced, not "Player: Alice" or "Level: 3"
    document.getElementById('score').textContent = 'Score: ' + points;
  }
</script>

aria-atomic="false" (the default for aria-live regions) announces only the mutated node(s). This avoids repeating static surrounding content every time one value changes. Useful when a region contains multiple independently-updating values.

11. aria-relevant — what mutations to announce

<!-- Default: "additions text" — new nodes + text changes -->
<div aria-live="polite" aria-relevant="additions text">...</div>

<!-- Removals only — e.g., items leaving a queue -->
<div aria-live="polite" aria-relevant="removals">...</div>

<!-- All mutations: additions, removals, and text changes -->
<div aria-live="polite" aria-relevant="all">...</div>

<!-- Additions and removals, but not text changes -->
<div aria-live="polite" aria-relevant="additions removals">...</div>

<!-- Practical example: notification queue -->
<div
  role="log"
  aria-relevant="additions removals"
  id="notif-queue"
></div>

aria-relevant controls which DOM mutation types trigger announcements: "additions" (new nodes), "removals" (removed nodes), "text" (text content changes), or combinations. The default is "additions text". Support varies — don't rely on "removals" in production.

12. Polite vs assertive — decision guide

<!-- USE POLITE (role="status" or aria-live="polite") for: -->
<div role="status">
  <!-- Form save confirmation -->
  <!-- Cart item count update -->
  <!-- Search results count -->
  <!-- Progress percentage -->
  <!-- Non-critical notifications -->
</div>

<!-- USE ASSERTIVE (role="alert" or aria-live="assertive") for: -->
<div role="alert">
  <!-- Session about to expire (60 seconds warning) -->
  <!-- Form submission failed (server error) -->
  <!-- Network connection lost -->
  <!-- Security/permission error -->
</div>

<!-- NEVER use assertive for: routine updates, optional info,
     or anything the user wasn't actively waiting for -->

Choosing the wrong politeness level degrades the experience. Assertive interrupts immediately — overusing it makes every action disruptive. When in doubt, choose polite. Reserve assertive for situations where a delayed announcement would cause genuine harm or confusion.

13. setTimeout pattern for deferred screen reader announcements

<div role="status" id="announce"></div>

<script>
  // Problem: setting textContent immediately after a focus change
  // can race with the screen reader's announcement of the focus event.

  // Solution 1: requestAnimationFrame (one frame delay)
  function announceRaf(text) {
    const el = document.getElementById('announce');
    el.textContent = '';
    requestAnimationFrame(() => { el.textContent = text; });
  }

  // Solution 2: small setTimeout (more reliable across AT)
  function announceSoon(text) {
    const el = document.getElementById('announce');
    el.textContent = '';
    setTimeout(() => { el.textContent = text; }, 100);
  }

  // After route change in SPA:
  announceSoon('Navigated to About page');
</script>

Screen readers can miss live region updates that happen in the same JS microtask as a focus change. A 100ms setTimeout (or rAF) decouples the announcement from the focus event, giving AT time to process the focus before the new text arrives.

INTERMEDIATE (14–26)

14. Form save notification

<form id="settings-form">
  <fieldset>
    <legend>Email preferences</legend>
    <label>
      <input type="checkbox" name="marketing">
      Marketing emails
    </label>
  </fieldset>
  <button type="submit">Save preferences</button>
</form>

<!-- Live region always in DOM -->
<div role="status" id="save-status" aria-atomic="true"></div>

<script>
  document.getElementById('settings-form').addEventListener('submit', async e => {
    e.preventDefault();
    const btn = e.target.querySelector('[type="submit"]');
    btn.disabled = true;
    btn.textContent = 'Saving…';

    await savePreferences(new FormData(e.target));

    btn.disabled = false;
    btn.textContent = 'Save preferences';
    document.getElementById('save-status').textContent =
      'Preferences saved successfully.';
  });
</script>

Pattern for save confirmations: disable the button while saving to prevent double-submit, restore it on completion, then announce success via role="status". Focus remains on the button, keeping the user oriented.

15. Error announcement on form submission

<form id="signup-form" novalidate>
  <label for="email">Email</label>
  <input type="email" id="email" name="email" aria-required="true">

  <button type="submit">Sign up</button>
</form>

<div role="alert" id="form-errors" aria-atomic="true"></div>

<script>
  document.getElementById('signup-form').addEventListener('submit', e => {
    e.preventDefault();
    const email = document.getElementById('email');
    const errors = [];

    if (!email.value) errors.push('Email is required.');
    else if (!email.validity.valid) errors.push('Please enter a valid email.');

    const errRegion = document.getElementById('form-errors');
    if (errors.length) {
      errRegion.textContent = errors.join(' ');
      email.setAttribute('aria-invalid', 'true');
      email.focus(); // Move focus to first error field
    } else {
      errRegion.textContent = '';
      submitForm();
    }
  });
</script>

On validation failure, use role="alert" for immediate announcement of the error summary, set aria-invalid on the offending field, then move focus to it. The combination of alert + focus ensures the user hears the error and lands in the right place.

16. Search results count announcement

<form role="search">
  <label for="product-search">Search products</label>
  <input type="search" id="product-search" name="q">
  <button type="submit">Search</button>
</form>

<div role="status" id="results-count" aria-atomic="true"></div>

<div id="results-grid">

</div>

Announcing search result counts via role="status" helps screen reader users understand the outcome of their search without navigating into the results first. Always include the query term for context and handle zero results explicitly.

17. Shopping cart count update

<header>
  <a href="/cart" aria-label="Cart" id="cart-link">
    <svg aria-hidden="true">...</svg>
    <span id="cart-count" class="badge">0</span>
  </a>
</header>

<!-- Separate live region — don't use the badge itself -->
<div role="status" class="sr-only" id="cart-announcement"></div>

<script>
  function addToCart(product) {
    const newCount = cartStore.add(product);

    // Update visual badge (silent)
    document.getElementById('cart-count').textContent = newCount;

    // Update accessible label on link
    document.getElementById('cart-link').setAttribute(
      'aria-label', `Cart, ${newCount} item${newCount === 1 ? '' : 's'}`
    );

    // Announce the action
    document.getElementById('cart-announcement').textContent =
      `${product.name} added to cart. Cart total: ${newCount} items.`;
  }
</script>

Separate the visual badge from the live region announcement. The badge number silently updates; the live region announces the action. Also update the cart link's accessible name so users can verify the count when tabbing to the header.

18. Spinner with accessible status

<!-- Spinner container -->
<div id="loading-container" hidden>
  <svg
    class="spinner"
    aria-hidden="true"
    focusable="false"
  >...</svg>
  <span class="sr-only">Loading...</span>
</div>

The animated spinner is decorative (aria-hidden). A separate static sr-only "Loading..." label is visible to AT from the spinner container. The live region handles state transitions, announcing both the start and completion of loading.

19. Progress bar with live region announcement

<label for="upload-progress">
  File upload progress
</label>
<progress
  id="upload-progress"
  value="0"
  max="100"
  aria-describedby="progress-status"
>0%</progress>

<div role="status" id="progress-status" aria-atomic="true">
  Upload not started.
</div>

<script>
  function updateProgress(percent) {
    const bar = document.getElementById('upload-progress');
    bar.value = percent;

    // Announce milestones, not every percent tick
    const status = document.getElementById('progress-status');
    if (percent === 100) {
      status.textContent = 'Upload complete!';
    } else if (percent % 25 === 0) {
      status.textContent = `Upload ${percent}% complete.`;
    }
  }
</script>

The native <progress> element has implicit role="progressbar". Pair it with a live region but only announce at meaningful milestones (25%, 50%, 75%, 100%) rather than every percent change, which would flood the screen reader output.

20. Notification toast system

<!-- Toast container — always in DOM -->
<div
  id="toast-region"
  role="status"
  aria-live="polite"
  aria-atomic="false"
  class="toast-container"
></div>

<script>
  let toastId = 0;

  function showToast(message, type = 'info', duration = 5000) {
    const region = document.getElementById('toast-region');
    const toast = document.createElement('div');
    toast.className = `toast toast--${type}`;
    toast.id = `toast-${++toastId}`;
    toast.textContent = message;
    toast.setAttribute('role', type === 'error' ? 'alert' : 'status');

    region.appendChild(toast); // Announced on append

    // Auto-dismiss
    setTimeout(() => {
      toast.remove(); // Removal not announced (aria-relevant default)
    }, duration);
  }

  showToast('File saved.', 'success');
  showToast('Connection error.', 'error');
</script>

Toast notifications use a persistent container with aria-live. Individual toasts are appended (triggering announcement) and removed silently. Error toasts get role="alert" for assertive announcement; others use role="status" for polite delivery.

21. Chat message log with live region

22. Countdown timer with role="timer"

<!-- role="timer" implies aria-live="off" by default -->
<div
  role="timer"
  aria-live="off"
  aria-label="Session timeout countdown"
  id="session-timer"
>
  05:00
</div>

<!-- Separate alert for critical threshold -->
<div role="alert" id="timeout-alert"></div>

<script>
  let seconds = 300;
  const timerEl = document.getElementById('session-timer');
  const alertEl = document.getElementById('timeout-alert');

  const interval = setInterval(() => {
    seconds--;
    const m = Math.floor(seconds / 60).toString().padStart(2, '0');
    const s = (seconds % 60).toString().padStart(2, '0');
    timerEl.textContent = `${m}:${s}`;

    // Announce at 60 seconds remaining
    if (seconds === 60) {
      alertEl.textContent = 'Warning: 1 minute until session timeout.';
    }
    if (seconds === 0) {
      clearInterval(interval);
      alertEl.textContent = 'Session expired. Please log in again.';
    }
  }, 1000);
</script>

role="timer" is for countdowns and elapsed time. Crucially, it has implicit aria-live="off" — the visual ticker updates without constant AT interruption. Critical thresholds trigger a separate role="alert" announcement at 60 seconds and on expiry.

23. File upload progress with live region

<input type="file" id="file-input" multiple accept="image/*">
<div id="upload-list"></div>
<div role="status" id="upload-summary" aria-atomic="true"></div>

<script>
  document.getElementById('file-input').addEventListener('change', async (e) => {
    const files = [...e.target.files];
    const summary = document.getElementById('upload-summary');
    summary.textContent = `Uploading ${files.length} file${files.length === 1 ? '' : 's'}.`;

    let completed = 0;
    await Promise.all(files.map(async file => {
      await uploadFile(file);
      completed++;
      summary.textContent =
        `${completed} of ${files.length} files uploaded.`;
    }));

    summary.textContent =
      `All ${files.length} files uploaded successfully.`;
  });
</script>

For multi-file uploads, update a single role="status" with an aggregate count rather than announcing each file individually. The atomic summary "3 of 5 files uploaded" is more useful than five separate announcements.

24. Server-sent events with live region

<div
  role="log"
  aria-label="Live server events"
  aria-live="polite"
  id="event-log"
></div>

<div role="status" id="connection-status">Connecting...</div>

<script>
  const es = new EventSource('/api/events');
  const log = document.getElementById('event-log');
  const status = document.getElementById('connection-status');

  es.onopen = () => {
    status.textContent = 'Connected to live updates.';
  };

  es.onmessage = (e) => {
    const data = JSON.parse(e.data);
    const item = document.createElement('p');
    item.textContent = `[${data.timestamp}] ${data.message}`;
    log.appendChild(item);
    // Trim log to last 50 messages
    while (log.children.length > 50) log.firstChild.remove();
  };

  es.onerror = () => {
    status.textContent = 'Connection lost. Attempting to reconnect.';
  };
</script>

Real-time event streams use role="log" for the feed and role="status" for connection state. Trim old messages to prevent infinite growth. The connection state changes (connected / lost) are announced via the polite status region.

25. SPA page transition announcement

<!-- Place in root layout; persists across route changes -->
<div
  role="status"
  id="route-announcer"
  class="sr-only"
  aria-atomic="true"
  aria-live="polite"
></div>

<script>
  // React Router example (apply the same pattern in any SPA framework)
  function announceRouteChange(location) {
    // Derive a friendly name from the route or document title
    const title = document.title || location.pathname;
    const announcer = document.getElementById('route-announcer');

    // Clear then set to ensure re-announcement
    announcer.textContent = '';
    setTimeout(() => {
      announcer.textContent = `Navigated to ${title}`;
    }, 100);
  }

  // In React Router v6:
  // useEffect(() => announceRouteChange(location), [location]);
</script>

SPAs change content without full page reloads, so screen readers don't get the native browser page-load announcement. A persistent route announcer div reads the new page title on every navigation, maintaining orientation for AT users.

NESTED (27–38)

27. Form validation with live feedback

<form novalidate>
  <div class="field">
    <label for="username">Username</label>
    <input
      type="text"
      id="username"
      aria-required="true"
      aria-describedby="username-status"
      autocomplete="username"
    >
    <!-- Live region for this field's status -->
    <div
      id="username-status"
      role="status"
      aria-live="polite"
      aria-atomic="true"
    ></div>
  </div>

  <button type="submit">Register</button>
</form>

<script>
  let checkTimer;
  document.getElementById('username').addEventListener('input', e => {
    clearTimeout(checkTimer);
    checkTimer = setTimeout(async () => {
      const val = e.target.value;
      const status = document.getElementById('username-status');
      if (val.length < 3) {
        status.textContent = 'Username must be at least 3 characters.';
        e.target.setAttribute('aria-invalid', 'true');
      } else {
        const taken = await checkAvailability(val);
        status.textContent = taken
          ? `"${val}" is already taken.`
          : `"${val}" is available.`;
        e.target.setAttribute('aria-invalid', taken ? 'true' : 'false');
      }
    }, 500);
  });
</script>

Per-field live regions provide real-time feedback with 500ms debounce to avoid flooding AT while typing. The region is described via aria-describedby so AT reads it on focus. Polite delivery avoids interrupting the typing experience.

29. Search-as-you-type results with live region

36. Announcement queue pattern

<div role="status" id="queue-region" aria-atomic="true"></div>

<script>
  const announceQueue = (() => {
    const queue = [];
    let isAnnouncing = false;
    const region = document.getElementById('queue-region');

    function processNext() {
      if (!queue.length) { isAnnouncing = false; return; }
      isAnnouncing = true;
      const msg = queue.shift();

      region.textContent = '';
      requestAnimationFrame(() => {
        region.textContent = msg;
        // Wait for AT to read (approximately 1.5s per message)
        setTimeout(processNext, 1500);
      });
    }

    return {
      add(message) {
        queue.push(message);
        if (!isAnnouncing) processNext();
      }
    };
  })();

  // Usage: multiple rapid events won't overlap
  announceQueue.add('Item A added to cart.');
  announceQueue.add('Discount applied: 10% off.');
  announceQueue.add('Free shipping unlocked!');
</script>

When multiple events fire rapidly, a queue prevents announcements from overwriting each other. The queue processes one message at a time with 1.5s gaps — enough for a screen reader to finish reading. Clear-then-set with rAF ensures each message triggers a fresh mutation.

50. Comprehensive live region reference summary

<!-- ROLE REFERENCE -->
<!--
  role="alert"    → assertive + atomic="true"   | Errors, urgent info
  role="status"   → polite   + atomic="true"   | Confirmations, status
  role="log"      → polite   + relevant="additions" | Logs, chat
  role="marquee"  → off                         | Promotional tickers
  role="timer"    → off                         | Countdowns

  ATTRIBUTES:
  aria-live       = "off" | "polite" | "assertive"
  aria-atomic     = "true" | "false"  (default: false)
  aria-relevant   = "additions" | "removals" | "text" | "all"
                    (default: "additions text")
  aria-busy       = "true" | "false"  (suppress while loading)

  GOLDEN RULES:
  1. Live regions must be in DOM on page load (never inject dynamically)
  2. Only one assertive region per page
  3. Use polite by default; assertive only for genuine emergencies
  4. Clear before set to re-announce identical text
  5. Never use display:none or visibility:hidden on a live region
  6. Debounce high-frequency updates
  7. Announce completion, not just start
  8. Test with NVDA, JAWS, VoiceOver, and Narrator
-->

Quick-reference summary of all live region roles, attributes, and golden rules. Print or bookmark this for accessibility reviews. The eight golden rules cover the most common implementation mistakes found in production applications.
