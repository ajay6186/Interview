Examples 1.5 — Forms Basic

BASIC (1–13)

1. Minimal form

Minimal form: action, method, label[for] matching input[id], and a submit button.

2. input type="text"

type="text" is the default. placeholder shows ghost text when empty — not a substitute for a label.

3. input type="email"

type="email" provides built-in validation and a custom keyboard on mobile devices.

4. input type="password"

type="password" masks the input. Always use autocomplete="current-password" or "new-password" to help password managers.

5. textarea

<textarea> is for multi-line text. rows and cols suggest initial size. CSS resize: vertical limits to vertical resize only.

6. button types

submit = submits the form. reset = clears all fields (rarely useful). button = no default action, use for JS hooks.

7. Label wrapping input (implicit association)

Wrapping <input> inside <label> creates an implicit association — no for/id needed. But explicit for/id is preferred.

8. input type="checkbox"

Checkboxes work best with wrapping labels for a larger click target.

9. Radio button group

Preferred contact method:

Radio buttons in a group must share the same name. Only one can be selected at a time.

10. select dropdown

The first option with an empty value acts as a placeholder. Always include an associated label.

11. input type="number"

type="number" shows a numeric keyboard on mobile and browser spin buttons. Use min/max/step to constrain values.

12. input type="date"

type="date" provides a native date picker in most browsers. Value format is always YYYY-MM-DD.

13. form with GET method

GET method appends form data to the URL (?q=term). Use for search forms and bookmarkable queries.

INTERMEDIATE (14–26)

14. input type="tel"

type="tel" shows a phone keypad on mobile but doesn't validate format. Use pattern attribute to enforce format.

15. input type="url"

type="url" requires a valid URL format (including protocol). Browsers show a URL keyboard on mobile.

16. input type="range"

type="range" renders a slider. Use <output> to display the current value alongside it.

17. input type="color"

type="color" opens a color picker. Value is always a hex color string (#rrggbb).

18. input type="file"

accept filters which file types the picker shows. For multiple files, add the multiple attribute. Requires enctype="multipart/form-data" on the form.

19. input type="hidden"

<input type="hidden" name="csrf_token" value="abc123xyz">
<input type="hidden" name="redirect_to" value="/dashboard">

Hidden inputs submit data without user interaction. Common for CSRF tokens, page state, and back-end tracking values.

20. required and placeholder

required prevents form submission if empty. Combine with a visual indicator (*) for sighted users and aria-required for screen readers.

21. autocomplete attribute

autocomplete tokens (name, email, street-address, etc.) help browsers and password managers fill in forms correctly.

22. autofocus

autofocus moves keyboard focus to this field on page load. Use only when the primary purpose of the page is the form.

23. readonly vs disabled

readonly: user can't change the value but it IS submitted. disabled: grayed out, not interactive, and NOT submitted.

24. Multiple select

multiple allows selecting several options. size controls how many rows are visible. Users hold Ctrl/Cmd to multi-select.

25. optgroup in select

<optgroup> groups related options under a non-selectable heading label.

26. datalist autocomplete

<datalist> provides autocomplete suggestions while allowing free text. Connect via list attribute matching datalist id.

NESTED (27–38)

27. Fieldset with legend

Shipping Address

<fieldset>/<legend> groups related controls. The legend is announced as the group label by screen readers.

28. Radio group inside fieldset

Subscription Plan

Radio groups should always be wrapped in <fieldset>/<legend> so screen readers announce the group question.

29. Multi-step form shell

<form id="signup-form">
  <!-- Step 1 -->
  <fieldset id="step-1">
    <legend>Step 1 of 3: Account Details</legend>
    <label for="s1-email">Email</label>
    <input type="email" id="s1-email" name="email">
  </fieldset>
  <!-- Steps 2 and 3 hidden initially via JS -->
</form>

Multi-step forms use fieldsets for each step. JS shows/hides steps. aria-current="step" can track progress.

36. Search form with role="search"

role="search" identifies the form as a search landmark. The visually-hidden label or aria-label names the input for screen readers.

ADVANCED (39–50)

39. form attribute to link input to external form

<form id="main-form" action="/submit" method="post">
  <label for="ex39-name">Name</label>
  <input type="text" id="ex39-name" name="name">
</form>

<!-- This input is OUTSIDE the form but still part of it -->
<input type="email" name="email" form="main-form">
<button type="submit" form="main-form">Submit</button>

The form attribute links any input/button to a form by ID, even if the element is outside the form in the DOM.

40. formaction and formmethod overrides

<form action="/save-draft" method="post">
  <input type="text" name="content">
  <button type="submit">Save Draft</button>
  <button type="submit" formaction="/publish" formmethod="post">Publish</button>
</form>

formaction and formmethod on a submit button override the form's action and method for that specific submission.

41. Custom validity with setCustomValidity

setCustomValidity() sets a custom browser validation message. Pass an empty string to clear it.

42. novalidate — disable browser validation

<form action="/submit" method="post" novalidate>
  <!-- Browser won't validate; use custom JS validation instead -->
  <input type="email" name="email" required>
  <button type="submit">Submit</button>
</form>

novalidate disables browser-native validation. Use when implementing custom validation UI that replaces the browser's default popups.

43. FormData API

<form id="ajax-form">
  <input type="text" name="username" value="Alice">
  <input type="email" name="email" value="alice@example.com">
</form>

<script>
  const form = document.getElementById("ajax-form");
  const data = new FormData(form);
  for (const [key, value] of data.entries()) {
    console.log(key, "=", value);
  }
  // Submit via fetch:
  fetch("/api/submit", { method: "POST", body: data });
</script>

FormData serialises a form's values (including files) for fetch/XMLHttpRequest submission without page navigation.

44. Constraint validation API

<input type="email" id="email-field" required>
<script>
  const field = document.getElementById("email-field");
  field.addEventListener("blur", () => {
    if (!field.validity.valid) {
      if (field.validity.valueMissing) showError("Email is required");
      else if (field.validity.typeMismatch) showError("Enter a valid email");
    }
  });
</script>

The ValidityState object (field.validity) exposes specific constraint failures: valueMissing, typeMismatch, patternMismatch, tooLong, etc.

45. Accessible error message pattern

<label for="email-acc">Email <span aria-hidden="true">*</span></label>
<input type="email" id="email-acc" name="email"
       required
       aria-required="true"
       aria-invalid="true"
       aria-errormessage="email-error">
<span id="email-error" role="alert">
  Please enter a valid email address.
</span>

aria-invalid="true" and aria-errormessage (referencing the error element's ID) communicate errors to screen readers without relying on visual cues alone.

46. Password confirmation validation

Cross-field validation via oninput: check confirm matches password using setCustomValidity.

47. Form with ARIA live region for status

<form id="contact-form">
  ...fields...
  <button type="submit">Send</button>
</form>
<div role="status" aria-live="polite" aria-atomic="true" id="form-status"></div>

<script>
  document.getElementById("contact-form").addEventListener("submit", async e => {
    e.preventDefault();
    await submitForm();
    document.getElementById("form-status").textContent = "Message sent successfully!";
  });
</script>

role="status" with aria-live="polite" announces form submission outcomes to screen reader users without disrupting reading flow.

48. input with pattern and title

pattern accepts a JavaScript RegExp (without slashes). The title attribute becomes the browser's validation message tooltip.
